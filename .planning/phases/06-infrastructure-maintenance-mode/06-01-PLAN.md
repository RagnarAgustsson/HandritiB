})

**proxy.ts changes:**
Insert maintenance mode check at the TOP of the clerkMiddleware callback, before the auth protect and i18n routing:

1. Read `process.env.MAINTENANCE_MODE` — only activate when exactly `"true"`
2. Skip maintenance mode for:
   - All `/api/*` routes (webhooks, cron, health must keep working)
   - The `/maintenance` path itself (avoid redirect loop)
   - Admin users: use `auth()` to get `sessionClaims`, check if user has admin role via `sessionClaims?.metadata?.role === 'admin'` OR query the admins table. Since middleware should be fast, check Clerk's `sessionClaims.publicMetadata.isAdmin` if available, otherwise fall back to checking `orgRole` or a simpler approach: let the middleware check if the user is signed in and has the admin claim. If admin metadata is not available in Clerk claims, use a simpler approach: check if `userId` exists in the `admins` table by importing `db` and `admins` from `lib/db`. However, DB queries in middleware add latency — prefer Clerk publicMetadata if possible. Use a pragmatic approach: check `process.env.ADMIN_SEED_EMAIL` against the session's email claim, or use Clerk's `sessionClaims?.publicMetadata` if the project sets admin metadata there. Look at how the existing admin panel checks admin status and mirror that pattern.
3. When maintenance mode is active and user is NOT admin: rewrite to `/maintenance` using `NextResponse.rewrite(new URL('/maintenance', request.url))`
4. When maintenance mode is active and user IS admin: continue normally (they see the real app)
